## 一、产品概述

### 1.1 一句话
把女朋友从 500 张照片里挑好片的 30 分钟变成 30 秒。

### 1.2 目标用户
爱拍照的女生。每次出行拍 100-500 张，拍完不想删也懒得筛。

### 1.3 产品定位
**本地桌面 App（macOS 优先），只管筛选，不做修图、不做相册、不上云。**

---

## 二、用户痛点

| 痛点 | 典型场景 |
|------|---------|
| 闭眼 | 合影 10 张 8 张有人闭眼，逐一放大检查 |
| 模糊 | 手抖糊片，缩略图根本看不出来 |
| 曝光灾难 | 背景过曝脸全黑 / 夜景噪点爆炸 |
| 构图翻车 | 人拍歪、裁到脚、头顶空间诡异 |
| 表情崩坏 | 翻白眼、打哈欠、面瘫瞬间 |
| 连拍地狱 | 同姿势 20 张，选不出留哪张 |
| 审美疲劳 | 看到后面分不清好坏，随机留几张 |

### 4.2 筛选规则细则

#### 第 1 层：客观缺陷（固定阈值，一票否决）

| 检测项 | 方法 | 阈值 | 为什么能固定 |
|--------|------|------|-------------|
| 闭眼 | EAR（眼部纵横比） | < 0.18 | 比值，与光线/分辨率无关 |
| 张嘴崩坏 | MAR（嘴部纵横比） | > 0.7 | 比值，纯几何量 |
| 严重歪斜 | 头部 roll 角 | > 20° | 纯几何量 |
| 人脸裁切 | 人脸框距画面边缘 | margin < 5% | 比例 |
| 人脸过小 | 人脸框 / 画面面积 | < 1% | 比例 |

> 以上任一命中 → 直接标记废片。用户可在 "已筛除" 列表中找回。

#### 第 2 层：主观质量（批次内相对排名）

| 维度 | 方法 | 判定逻辑 |
|------|------|---------|
| 模糊 | 批次内拉普拉斯方差排序 | 后 20%（即最模糊的 20%）标记为"清晰度偏低" |
| 曝光 | 批次内 HSV-V 亮度中位偏离 | 最暗 15% 标记"偏暗"；最亮 15% 标记"偏亮" |
| 美学 | NIMA 模型打分（1-10 分） | 前 30% 的 NIMA 高分加权计入综合排序 |

> 不设绝对阈值。同一批照片共享拍摄条件，**批次内最差的就是应该筛掉的**。

#### 第 3 层：连拍去重

- pHash 计算图片指纹，汉明距离 < 5 视为相似
- SSIM 二次确认，排除误匹配
- 相似组内按第 2 层综合评分排序，保留 Top 2
- 用户可在设置中调整每组保留张数

## 五、综合评分算法

### 5.1 公式

```
总分（0-1） = 模糊排名 × 0.25 + 曝光排名 × 0.15 + NIMA美学 × 0.30 + 表情趋势 × 0.20 + 人脸清晰度 × 0.10
```

- 模糊排名：批次内拉普拉斯方差百分位，归一化到 0-1（1 = 最清晰）
- 曝光排名：批次内亮度中位接近度，归一化到 0-1（1 = 最接近批次中位数）
- NIMA 美学：NIMA 模型 1-10 分除以 10
- 表情趋势：嘴角上扬角度正面 + MAR 逆向，归一化到 0-1（1 = 微笑且不张嘴）
- 人脸清晰度：人脸区域局部拉普拉斯方差，批次内归一化到 0-1

### 5.2 废片红线（不参与排名）

| # | 条件 | 含义 |
|---|------|------|
| 1 | EAR < 0.18 | 闭眼 |
| 2 | 人脸占比 < 1% | 几乎看不见人 |
| 3 | 头部 roll > 20° | 严重歪斜 |
| 4 | 人脸距边缘 < 5% | 主体被裁切 |
| 5 | MAR > 0.7 | 张嘴过大 |

> 仅 5 个纯几何量做一票否决。模糊/曝光/构图不做一票否决——它们的 "坏" 的标准随场景变化。

## 六、用户操作流程

### 6.1 主流程

```
打开 App → 拖入文件夹 → 等待分析（进度条）→ 浏览精选结果 → 手势微调 → 一键导出
```

| 步骤 | 动作 | 系统反馈 | 目标耗时 |
|------|------|---------|---------|
| 1 | 启动 App | 空白页 + "拖入照片文件夹开始" | < 1s |
| 2 | 拖入文件夹 | 扫描进度 + 已找到照片数 | < 5s |
| 3 | 确认开始 | 实时进度条 + "正在认真看每一张 ✨" | 500 张 < 90s |
| 4 | 浏览结果 | 默认展示 Top 30 精选，大图 + 分维标签 | 即时 |
| 5 | 手势复核 | 左右滑动，实时更新精选/筛除 | 2-5 min |
| 6 | 导出 | 选择路径 + 自动命名 + 完成弹窗 | < 10s |

### 6.2 异常处理

| 异常 | 处理 |
|------|------|
| 文件夹无图片 | "没找到支持的图片（jpg/png/heic）" |
| 照片 < 5 张 | 跳过 AI，直接进入浏览 |
| 照片 > 1000 张 | "照片较多（X 张），预计 Y 分钟，继续？" |
| 分析中途关闭 | 自动保存进度，下次可继续 |
| 程序崩溃 | 已分析结果全保留，原始文件不受影响 |

### 8.2 核心架构决策：为什么不自己训模型 / 不用大模型

#### 决策框架

```
┌───────────────────────────────────────────────────────────────┐
│                    PhotoForYou 分析流水线                       │
│                                                               │
│  第 1 层：客观缺陷（固定阈值）                                  │
│  ┌──────────┬──────────┬──────────┬──────────┬──────────┐    │
│  │ 闭眼 EAR │ 张嘴 MAR │ 歪斜 Roll│ 裁切距离 │ 过小占比 │    │
│  │  < 0.18  │  > 0.7   │  > 20°   │ margin<5%│  < 1%    │    │
│  └──────────┴──────────┴──────────┴──────────┴──────────┘    │
│  依赖：MediaPipe Face Mesh（一次推理 30ms）                     │
│                                                               │
│  第 2 层：主观质量（批次内排名 + NIMA）                         │
│  ┌──────────────────────────────────────────────────────┐    │
│  │  模糊 → 批次内拉普拉斯方差百分位，后 20% 降权         │    │
│  │  曝光 → 批次内亮度中位偏离度，首尾 15% 降权           │    │
│  │  美学 → NIMA MobileNetV2 打分（ONNX 推理 50ms/张）    │    │
│  └──────────────────────────────────────────────────────┘    │
│  依赖：OpenCV.js + ONNX Runtime（Worker 内执行）              │
│                                                               │
│  第 3 层：去重                                                │
│  ┌──────────────────────────────────────────────────────┐    │
│  │  pHash 聚类（汉明距离 < 5）→ SSIM 二次确认 → 组内按  │    │
│  │  第 2 层综合排名保留 Top 2                             │    │
│  └──────────────────────────────────────────────────────┘    │
└───────────────────────────────────────────────────────────────┘
```

#### 为什么不用自己训练的神经网络

| 原因 | 说明 |
|------|------|
| 无标注数据 | 没有成百上千张 "好/坏" 标签的照片，不可能训监督模型 |
| 任务已解决 | 闭眼↔EAR、张嘴↔MAR、美学↔NIMA，无需自己造轮子 |
| 部署成本高 | 自训模型需 PyTorch → ONNX 导出 → 绑 Electron，模型文件动辄 100MB+ |
| GPU 依赖 | 目标设备 MacBook Air 无独显，自训网络推理慢 |

#### 为什么 NIMA 不需要自己训练

Google 的 NIMA（Neural Image Assessment）在 **AVA 数据集**（Aesthetic Visual Analysis，25 万张照片，每张有专业摄影师 1-10 分评分）上预训练完毕。核心参数：

- 骨干网络：MobileNetV2（专为移动端设计，轻量）
- 模型大小：~14MB
- 推理速度：~50ms/张（CPU）
- 标注规模：25 万张 × 多位评分者 = 覆盖晴天/夜景/逆光/室内/风景/人像/宠物/食物等全场景
- 部署方式：下载预训练权重 → 转 ONNX → onnxruntime-node 加载 → 本地推理

**NIMA 补了纯规则永远覆盖不了的那块**：一张逆光人像，NIMA 知道它不是过曝，是有意为之；一张夜景，NIMA 知道暗部是正常的，不是欠曝。

#### 为什么不全用 NIMA

NIMA 输出一个整体分数，不告诉你**为什么**扣分。产品需要给用户可解释的标记（"闭眼了"、"手抖了"），而不是一个神秘的数字。所以 NIMA 只用于构图/美学维度，其他维度保持可解释。

#### 为什么不用 GPT-4V / Claude Vision（生产环境）

| 对比 | 大模型 API | 本方案 |
|------|-----------|-------|
| 延迟 | 2-5s / 张 | 90-150ms / 张 |
| 500 张成本 | ~$10 | $0 |
| 隐私 | 照片上传第三方 | 全程本地 |
| 离线 | ❌ | ✅ |
| 确定性 | 可能波动 | 数学结果，可复现 |
| 可调性 | 改 prompt | 每个阈值精确可调 |

> **V1.5 预留**：用户可对 "待定区" 边缘 20 张调一次大模型做 "第二意见"——20 次调用成本和延迟都可控。但 MVP 不做。

### 8.3 海量图片加载策略

#### 8.3.1 三层缓存架构

```
                    文件系统（磁盘）
                 500 张 .jpg，共 2-5 GB
                          │
            ┌─────────────▼─────────────┐
            │  L1：元数据索引（常驻）   │
            │  路径 + 分辨率 + EXIF    │
            │  500 张 ≈ 100 KB        │
            └─────────────┬─────────────┘
                          │
     ┌────────────────────┼────────────────────┐
     ▼                    ▼                    ▼
┌──────────┐      ┌──────────────┐      ┌──────────┐
│ 缩略图   │      │ 分析结果     │      │ 原图预加载│
│ 400px    │      │ 每张分维评分 │      │ 当前±7张  │
│ WebP     │      │ 500≈200KB   │      │ 15张≈120MB│
│ 50MB     │      │             │      │ LRU 驱逐  │
└──────────┘      └──────────────┘      └──────────┘
```

#### 8.3.2 导入：两遍扫描

| 遍次 | 做什么 | I/O | 耗时（500 张） |
|------|--------|-----|--------------|
| 第一遍 | 遍历文件，读 EXIF + 分辨率头（2KB/张） | 读 2KB | < 1s |
| 第二遍 | sharp 并行转 400px WebP 缩略图 | 全量读 → 小输出 | ~15s（8 线程） |

第二遍完成即刻可浏览。分析在后台 Worker 线程池异步跑，不阻塞 UI。

#### 8.3.3 分析：Worker 线程池 + 流水线

```
主线程（React UI）
    │  分发任务 { filePath, thumbnailPath }
    ▼
┌───────────────────────────────────────────────┐
│              Worker Pool（8 线程）            │
│                                               │
│  单 Worker 流水线：                           │
│    sharp 缩放 800px（5ms）                    │
│    ↓                                          │
│    并行执行两组：                             │
│      A. OpenCV.js：拉普拉斯 + 直方图（10ms）  │
│      B. MediaPipe Face Mesh（30ms）          │
│      C. NIMA ONNX 推理（50ms）               │
│    ↓                                          │
│    计算分数 + 更新队列排名                     │
│    ↓                                          │
│    postMessage 回主线程 → 更新进度条 + 排名   │
│                                               │
│  8 线程并行，bottleneck 是 NIMA 的 50ms       │
│  500 张 ÷ 8 × 60ms ≈ 3.8s 理论值             │
│  实际约 30-60s（含 I/O 和调度开销）            │
└───────────────────────────────────────────────┘
```

- 分析用 800px 缩略图，不需要原分辨率（MediaPipe 和 NIMA 输入都要求小尺寸）
- Worker 用 `comlink` 库封装通信，避免手写 `postMessage` 样板代码

#### 8.3.4 为什么 Worker + WASM 而不是 Node 原生

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 主线程 OpenCV.js | 简单 | 阻塞 UI | ❌ |
| Node 子进程 sharp | 快 | IPC 序列化开销大、sharp 不能做人脸/美学 | ❌ |
| **Worker + OpenCV.js + ONNX** | 不卡 UI + 自然并行 | Worker 通信代码多（comlink 可解决） | ✅ |

### 8.4 丝滑浏览方案

#### 8.4.1 核心原理

始终只保留 3 个 DOM 节点（N-1、N、N+1），切换是 CSS `transform` 平移，不增删 DOM。前后各预加载 7 张原图到内存。

```
已浏览（释放）  ←  [7 张缓存 | 当前 N | 7 张预加载]  →  待浏览（只有缩略图）
                   ← 最大 15 张原图常驻 →
```

#### 8.4.2 渐进式图片加载

```
t=0ms     t=16ms       t=100ms         t=300ms
  │         │            │               │
  ▼         ▼            ▼               ▼
┌────┐  ┌──────┐   ┌─────────┐    ┌──────────┐
│色块│  │缩略图│   │ 中分辨率 │    │   原图   │
│占位│  │10KB  │   │  200KB   │    │   5MB    │
└────┘  └──────┘   └─────────┘    └──────────┘
 不空白   可辨认      可审片          可放大
```

实现：`dominantColor` 占位色块 → 缩略图 `<img>`（已缓存在内存）→ 中分辨率 `<canvas>`（lazy decode）→ 原图 `<img>`（lazy load，覆盖最上层）。

#### 8.4.3 手势滑动

三张 `<div>` 在 DOM 里（N-1/N/N+1），切换用 CSS `transform: translateX()` + `opacity`：
- 手指滑动 → 跟随手势实时平移（60fps）
- 松手距离 < 30% 窗口 → 回弹到原位
- 松手距离 ≥ 30% 窗口 → 动画吸附到下一张

#### 8.4.4 内存管理

```
LRU 驱逐窗口：最多 15 张原图
  驱逐 ← ← ← ← [15 张] ← ← ← ← 预加载

  15 × 5MB = 75MB  原图
+ 50MB            缩略图（WebP）
+ 50MB            React + Electron 基础
+ 14MB            ONNX 模型常驻
─────────────────────────
≈ 190MB           常驻内存
```

#### 8.4.5 感知性能对比

| 操作 | 不良体验 | PhotoForYou 目标 |
|------|---------|-----------------|
| 导入 500 张 | 等 10s+ | < 3s 可见首张缩略图 |
| 分析中 | 进度条卡死 | 实时百分比 + 每完成一张刷新排名 |
| 左右滑 | 白屏闪烁 | CSS transform 60fps |
| 双击放大 | 等 1s | 预加载缓存命中 < 100ms |
| 快滑 50 张 | 内存暴涨 | LRU 驱逐，内存曲线平坦 |

## 附录 A：术语表

| 术语 | 全称 | 解释 |
|------|------|------|
| EAR | Eye Aspect Ratio | 眼部纵横比，6 个关键点围成矩形的宽高比，< 0.18 = 闭眼 |
| MAR | Mouth Aspect Ratio | 嘴部纵横比，> 0.7 = 张嘴过大 |
| NIMA | Neural Image Assessment | Google 的美学评分模型，MobileNetV2 变体，AVA 数据集训练 |
| pHash | Perceptual Hash | 感知哈希，视觉相似图片产生相近哈希值 |
| SSIM | Structural Similarity Index | 结构相似度，补充 pHash 的二次确认 |
| ONNX | Open Neural Network Exchange | 模型交换格式，支持跨框架部署 |
| LRU | Least Recently Used | 最近最少使用缓存淘汰策略 |
